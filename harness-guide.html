Claude Harness Template

Claude Code 세션을 묶어 다단계 구현을 자동 실행하는 사내 표준 하네스. 이 자료는 팀 신규 인원이 처음부터 끝까지 따라할 수 있도록 만든 강의 문서다.

대상: 백엔드/프론트엔드 개발자 선수 지식: git, Python 3, Claude Code CLI 소요: 약 45분

01왜 이 하네스를 쓰는가

단일 LLM 호출로 큰 기능을 만들면 토큰이 터지고 일관성이 깨진다. 하네스는 작업을 잘게 쪼개고, 매 단계마다 가드레일과 검증을 강제한다.

해결하는 문제

  • 대형 기능을 한 세션에 욱여넣어 컨텍스트가 무너지는 문제
  • 중간에 실패해도 어디서 끊겼는지 추적이 안 되는 문제
  • 아키텍처 규칙(CLAUDE.md)을 LLM이 잊어버리는 문제
  • "잘 됐는지" 검증 없이 다음 단계로 넘어가는 문제

제공하는 것

  • Step 단위 분할 실행 — 한 step = 한 모듈/레이어
  • 가드레일 자동 주입 — CLAUDE.md + docs/*.md
  • 자가 교정 — 실패 시 최대 3회 재시도, 이전 에러 피드백
  • 상태 머신 — pending/completed/error/blocked 추적
  • 2단계 커밋 — 코드와 메타데이터를 분리
한 줄 요약. "사람이 step.md 잘 써두면, Python 스크립트가 Claude를 여러 번 깨워서 알아서 만들고 검증하고 커밋한다."

02멘탈 모델

하네스의 본질은 설계는 사람이, 실행은 LLM이, 검증은 둘 다. 강의 내내 이 그림을 머리에 두고 가자.

         사람                       Claude Harness                    Claude Code
         ────                       ───────────────                   ────────────

  PRD / ARCHITECTURE / ADR  ──▶
                                  guardrails 합쳐서 주입  ──▶  step.md 읽고 작업
       step{N}.md 작성     ──▶
                                  contract/summary 누적  ──▶  AC 실행 → 자가 검증
  index.json 초안 작성     ──▶                            ◀──  status 업데이트
                                  실패 시 에러 피드백     ──▶
                                  자동 commit (feat+chore)
  • 사람의 일 — PRD/ARCHITECTURE/ADR 문서화, CLAUDE.md 함정 누적, step.md 작성, AC 정의.
  • 하네스의 일 — 브랜치 관리, 가드레일 주입, 컨텍스트 누적, 재시도, 커밋.
  • Claude의 일 — step.md 지시대로 코드 작성, AC 실행, index.json 상태 업데이트.

04CLAUDE.md 작성법 — 카파시 + Anthropic 권장

하네스의 가드레일 중 절반의 무게가 CLAUDE.md에 실린다. 이 파일을 잘못 쓰면 하네스를 아무리 돌려도 결과가 어긋난다. 안드레 카파시(Andrej Karpathy)의 65줄 CLAUDE.md와 Anthropic 공식 작성 원칙을 종합한 가이드.

본질: README가 아니라 가드레일

흔히 하는 오해 — "CLAUDE.md = 프로젝트 설명서". 틀렸다. 이건 사람이 읽는 문서가 아니라 AI가 같은 실수를 반복하지 않도록 거는 행동 지침이다. 모든 줄이 토큰이라 매 호출마다 비용이 든다는 점도 기억하라.

안 쓸 것 vs 꼭 쓸 것

안 쓸 것 (안티패턴)꼭 쓸 것
"Next.js 15 + TypeScript + Tailwind 사용" "Tailwind arbitrary value (text-[12px]) 금지. 디자인 토큰만. 이유: 디자인 시스템 일관성"
"src/ 아래 app/components/lib/services" "API 호출은 무조건 src/services/ 경유. 클라이언트 컴포넌트에서 직접 fetch 금지. 이유: SSR 캐시 우회 사고"
"테스트는 잘 작성해주세요" "테스트는 vitest + msw 사용. render(<Component/>) 후 screen.getByRole 패턴. snapshot 금지."
"파일 구조를 잘 지키세요" "db.ts에 ORM 모델 추가 금지. 기존 5개 외에는 별도 파일. 이유: 모듈 간 결합도 폭증"

핵심 원칙: 코드 보면 알 수 있는 정보는 적지 마라. AI는 코드를 직접 읽을 수 있다. 적어야 할 건 코드만 봐선 절대 모를 함정·금지·도메인 트랩뿐이다.

카파시 4원칙 — 글로벌 CLAUDE.md에 권장

모든 프로젝트에 공통으로 적용할 일반 코딩 원칙은 ~/.claude/CLAUDE.md(개인 글로벌)에 두는 게 깔끔하다. 안드레 카파시가 정리한 4가지가 사실상 표준으로 자리잡았다.

① Think Before Coding

가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 표면화하라. 불명확하면 멈추고 물어라.

② Simplicity First

요청 범위 초과 기능 추가 금지. 일회용 코드 추상화 금지. 불가능한 시나리오 대비 에러 처리 금지. 200줄을 쓰면 50줄로 줄여라.

③ Surgical Changes

인접 코드 "개선" 금지. 깨지지 않은 거 리팩터 금지. 기존 데드 코드는 언급만 하고 삭제 금지. 기존 스타일 유지.

④ Goal-Driven Execution

단계별 지시 대신 성공 기준을 줘라. "검증 추가" → "잘못된 입력 테스트 작성, 통과 만들기". 강한 성공 기준은 AI가 독립적으로 루프 돌게 한다.

출처: forrestchang/andrej-karpathy-skills (GitHub 22만 star).

Anthropic 공식 작성 4원칙

원칙의미
구체성"테스트 잘 짜라" 금지. 어떤 도구(vitest), 어떤 패턴, 무엇을 피해야 하는지까지 박아라.
구조화된 마크다운AI가 스캔하기 쉽게 헤더·리스트·인덴트를 활용. 산문 단락은 비효율.
정기 검토안 쓰는 규칙은 지운다. 새 실수는 즉시 반영한다. 코드처럼 유지보수하는 살아있는 문서.
간결함 (100줄 이하)100줄 넘으면 coding-style.md 등으로 분리하고 CLAUDE.md엔 참조 한 줄만.

최고 ROI 워크플로우: "실수 즉시 반영"

가장 중요한 한 가지. 처음부터 완벽한 CLAUDE.md를 쓰려 하지 마라. Claude가 실수하는 바로 그 순간 이렇게 시켜라: 
"이런 실수는 다시 하지 마. 이걸 CLAUDE.md에 반영해줘."
이 한 문장이 카파시·Anthropic 공식 가이드 모두가 강조하는 핵심 워크플로우다. CLAUDE.md는 사고 일지처럼 누적시켜 키운다.

이 작업을 자동으로 해주는 도구가 이미 Claude Code에 들어 있다:

  • /claude-md-management:revise-claude-md — 현 세션에서 Claude가 한 실수·배운 점을 자동 분석해 CLAUDE.md 업데이트 안을 제시한다.
  • /claude-md-management:claude-md-improver — 프로젝트 내 모든 CLAUDE.md 파일을 점수화(80점/A-/B-)하고 리팩터 방향을 제안한다.

왜 명령어를 명시해야 하나 (토큰 비용)

빌드/테스트/린트 명령어를 CLAUDE.md에 안 적으면 Claude는 매번 package.json을 뒤지면서 명령어를 추측한다. 추측이 틀리면 다시 시도한다. 이게 모이면 토큰·시간·돈이 다 새어나간다. 짧은 한 블록이 전체 비용을 크게 줄인다.

## 명령어
npm run dev      # 개발 서버
npm run build    # 프로덕션 빌드
npm run lint     # ESLint
npm run test     # 테스트 (vitest)

6항목 체크리스트

  • 100줄 이하로 유지되고 있는가?
  • 빌드 · 테스트 · 린트 명령어가 명시되어 있는가?
  • 코드로 파악 가능한 불필요한 정보(기술 스택 나열 등)가 빠져 있는가?
  • 규칙이 추상적이지 않고 구체적인가? ("X 하지 마라. 이유: Y" 형식)
  • 한 달 단위로 안 쓰는 규칙을 정리하고 있는가?
  • Claude가 실수할 때마다 즉각 반영하고 있는가?
현재 템플릿의 한계. 이 저장소의 CLAUDE.md는 "기술 스택" 같은 placeholder 섹션을 안내하는데, 이는 카파시·Anthropic 권장에 맞지 않는다. 실제 프로젝트에 적용할 땐 기술 스택 섹션은 삭제하고 "함정/금지" 섹션으로 대체하라.

새 기능을 추가할 때 따르는 5단계. .claude/commands/harness.md를 그대로 가져온 것이다.

B

A. 탐색

/docs/ 하위 문서를 읽고 프로젝트의 기획·아키텍처·설계 의도를 파악한다. Claude에게 시킬 때는 Explore 에이전트를 병렬로 돌리면 빠르다.

B. 논의

구현을 위해 구체화해야 할 결정 사항이 있으면 코딩하기 전에 사용자와 합의한다. 예: "외부 API 결과를 캐시할 건가? TTL은?" — 이 단계를 건너뛰면 step.md에 잘못된 시그니처가 박힌다.

C. Step 설계

합의된 내용을 기반으로 step 초안을 만든다. 다음 섹션의 7원칙을 따른다.

D. 파일 생성

사용자가 승인하면 phases/ 아래에 실제 파일을 만든다. 생성 파일 3종 참조.

E. 실행

python3 scripts/execute.py 0-mvp           # 순차 실행
python3 scripts/execute.py 0-mvp --push    # 실행 후 브랜치 push

06Step 설계 7원칙

잘 쓴 step.md = 잘 굴러가는 하네스. 못 쓴 step.md = 3번 재시도 후 error. 이 원칙들은 협상 대상이 아니다.

#원칙의미
1Scope 최소화한 step에서 한 레이어/모듈만. 모듈 2개를 동시에 건드리면 step을 쪼개라.
2자기완결성각 step.md는 독립된 Claude 세션에서 실행된다. "이전에 얘기한 것처럼" 같은 외부 참조는 금지.
3사전 준비 강제"읽어야 할 파일" 섹션에 관련 문서와 이전 step 결과물 경로를 명시.
4시그니처 수준 지시함수/클래스 인터페이스만 제시하고 구현은 LLM 재량. 단, 보안·멱등성 같은 핵심 규칙은 박아넣는다. (= 카파시 Goal-Driven Execution)
5AC는 실행 가능한 커맨드"동작해야 한다"가 아니라 npm run build && npm test 같은 실행 커맨드.
6주의사항은 구체적으로"조심해라" → 금지. "X 하지 마라. 이유: Y" → 허용.
7네이밍step name은 kebab-case slug. 예: project-setup, api-layer, auth-flow.
카파시 철학과의 연결. 원칙 4·5·6은 카파시의 Goal-Driven Execution / Simplicity First / 구체성 원칙을 step 단위로 옮긴 것이다. step.md를 잘 쓰는 것이 곧 카파시 가이드를 적용하는 것이다.
가장 흔한 실패. 원칙 5를 무시하고 AC를 "기능이 잘 동작함" 같이 추상적으로 적는 케이스. LLM은 자기 자신이 잘 동작한다고 우긴 뒤 completed 처리한다. 반드시 실행 커맨드를 박아라.

07생성 파일 3종

step 설계가 끝나면 다음 3종 파일을 만든다. 형식은 엄격하다 — execute.py가 직접 파싱한다.

D-1. phases/index.json — 전체 현황

여러 task(=phase)를 묶는 top-level 인덱스. 새 task가 생길 때마다 phases 배열에 추가한다.

{
  "phases": [
    { "dir": "0-mvp", "status": "pending" }
  ]
}

D-2. phases/{task}/index.json — task 상세

{
  "project": "my-app",
  "phase": "0-mvp",
  "steps": [
    { "step": 0, "name": "project-setup", "status": "pending" },
    { "step": 1, "name": "core-types",    "status": "pending" },
    { "step": 2, "name": "api-layer",     "status": "pending" }
  ]
}
필드의미주의
project프로젝트명CLAUDE.md와 일치
phasetask 이름디렉토리명과 동일
steps[].step순번0부터 시작
steps[].namekebab-case slugstep.md 파일명과는 별개
steps[].status초기 "pending"execute.py가 갱신

execute.py가 실행 중 자동으로 채우는 필드는 절대 손으로 미리 넣지 마라:

  • created_at — task 시작 시 1회
  • started_at — 각 step 시작 시
  • completed_at / failed_at / blocked_at — 상태 전이 시

D-3. phases/{task}/step{N}.md — 각 step 1개씩

# Step 1: core-types

## 읽어야 할 파일

먼저 아래 파일들을 읽고 프로젝트의 아키텍처와 설계 의도를 파악하라:

- /docs/ARCHITECTURE.md
- /docs/ADR.md
- src/lib/db.ts          # step 0에서 생성됨

이전 step에서 만들어진 코드를 꼼꼼히 읽고, 설계 의도를 이해한 뒤 작업하라.

## 작업

src/types/user.ts에 다음 타입을 정의한다.

  export interface User {
    id: string;
    email: string;
    createdAt: Date;
  }

  export type UserCreateInput = Omit<User, "id" | "createdAt">;

핵심 규칙:
- id는 UUID v4. crypto.randomUUID() 사용. 외부 라이브러리 금지.
- email은 소문자 정규화. RFC 검증은 다음 step에서.

## Acceptance Criteria

  npx tsc --noEmit       # 타입 에러 없음
  npm test -- types      # types 테스트 통과

## 검증 절차

1. 위 AC 커맨드 실행
2. 아키텍처 체크리스트
3. index.json 해당 step 업데이트

## 금지사항

- uuid 패키지를 설치하지 마라. 이유: ADR-002 외부 의존성 최소화 원칙.
- 기존 테스트를 깨뜨리지 마라

08contract vs summary

최근 적용된 변경(HARNESS_CHANGE_A_APPLIED.md)으로 도입된 핵심 개념. 토큰을 아끼면서 품질은 유지하는 방법.

step이 completed 될 때 LLM은 두 필드를 함께 채운다:

필드대상길이다음 step prompt 주입
summary사람이 읽음제한 없음contract가 없을 때만 fallback
contract다음 step LLM이 읽음1~3줄항상 우선

실제 예시 (HARNESS_CHANGE_A_APPLIED.md 발췌):

{
  "step": 7,
  "name": "rule-engine",
  "status": "completed",
  "summary": "approved YAML rule만 ERROR/WARN을 만들 수 있는 rule engine 구현.
              paths.py에 dotted-path resolver 추가. eval/exec/compile 사용 금지.
              tests/unit/test_engine.py에 12개 케이스, tests/unit/test_paths.py에 8개
              케이스 통과. 설계 근거: ADR-003 참조...",
  "contract": "engine.execute_rules(rules_dir, model)→dict 추가.
               paths.resolve(model, dotted_path)→Any 추가.
               불변식: 승인된 YAML rule만 ERROR/WARN 생성, eval/exec 금지 유지."
}
왜 둘 다 필요한가. summary는 사람이 PR 검토하거나 사후에 "여기서 뭔 일이 있었지?" 추적할 때 본다. contract는 다음 step Claude에게 매번 prompt에 들어가므로 짧고 정확해야 한다 — 누적되면 토큰을 많이 잡아먹는다.

09execute.py가 해주는 것

우리가 손으로 안 해도 되는 일들. 한 번씩만 이해하고 잊어라.

1. 브랜치 관리

feat-{phase} 브랜치를 자동 checkout. 이미 있으면 이어 받는다.

2. 가드레일 주입

매 step prompt 앞에 CLAUDE.md + docs/*.md 전체를 합쳐서 넣는다.

3. 컨텍스트 누적

완료된 step의 contract (or summary)를 다음 prompt 머리에 붙인다.

4. 자가 교정

실패 시 최대 MAX_RETRIES=3회 재시도. 이전 error_message를 다음 시도 prompt에 그대로 전달.

5. 2단계 커밋

코드 변경 → feat(phase): step N — name. 메타데이터(index.json, step-output.json) → chore(phase): step N output.

6. 타임스탬프

started_at, completed_at, failed_at, blocked_at을 KST로 자동 기록.

매 step에서 Claude에게 전달되는 prompt 구조

┌─ 시스템 안내 (project, 작업 규칙)
├─ 가드레일 (CLAUDE.md 전문 + docs/*.md 전문)
├─ 이전 Step 산출물 (completed step들의 contract or summary)
├─ ⚠ 이전 시도 실패 (retry일 때만, 직전 error_message)
└─ 현재 step.md 전문
중요. 가드레일이 매번 풀로 들어가기 때문에 docs/와 CLAUDE.md에 쓸데없이 긴 내용을 넣으면 모든 step의 토큰이 비싸진다. 카파시 권장 100줄을 가능한 한 지켜라.

10에러 / blocked 복구

하네스가 멈췄을 때 무엇을 보고, 무엇을 고치고, 어떻게 재실행하는가.

상태 전이

상태의미다음 행동
pending아직 실행 안 됨execute.py 실행 대기
completedAC 통과다음 step으로 진행
error3회 재시도 후 실패사람이 원인 파악, pending으로 되돌리기
blocked사람 개입 필요 (API 키, 외부 인증 등)막힌 이유 해결 후 pending으로 되돌리기

error 복구 절차

  1. phases/{task}/index.json에서 해당 step의 error_message를 읽는다.
  2. 코드와 step.md를 다시 보고 무엇이 잘못됐는지 진단한다.
    • step.md가 모호하다 → step.md를 고친다.
    • AC가 비현실적이다 → AC를 고친다.
    • 실제 코드 버그다 → 직접 고치거나, 더 구체적으로 step.md를 다시 쓴다.
    • 같은 실수가 반복된다 → CLAUDE.md에 함정 노트로 박는다.
  3. 해당 step의 status를 "pending"으로 바꾸고 error_message, failed_at을 지운다.
  4. python3 scripts/execute.py {task} 재실행.

blocked 복구 절차

  1. blocked_reason 확인.
  2. 실제 원인 해결 (API 키 설정, 외부 서비스 가입, 수동 마이그레이션 등).
  3. status를 "pending"으로 되돌리고 blocked_reason, blocked_at 삭제.
  4. 재실행.
하지 마라. error/blocked 상태를 그대로 두고 새 step.md를 만들거나, completed로 거짓 마킹해서 다음으로 넘어가는 것. 하네스의 핵심은 단계마다 검증이다. 우회하면 의미가 없다.

12자주 하는 실수

팀에 와서 처음 1~2주에 거의 모두가 한 번씩 밟는 함정들.

실수왜 안 되는가올바른 방향
step 하나에 백엔드+프론트+테스트 다 넣기 한 세션에 너무 많은 컨텍스트가 들어가서 LLM이 우선순위를 잃는다 레이어/모듈별로 step을 쪼개라
AC를 "잘 동작해야 한다"로 적기 LLM이 자기 자신이 잘 동작한다고 우김 실행 커맨드 (npm test 등)로 적어라
"이전 step에서 만든 그거" 같은 외부 참조 step.md는 독립 세션에서 실행됨 — 그 컨텍스트가 없다 경로를 명시 (src/types/user.ts)
CLAUDE.md에 기술 스택·폴더 구조 나열 코드 보면 알 수 있는 정보 — 토큰 낭비 함정·금지·도메인 트랩만 남겨라
CLAUDE.md를 처음부터 완벽하게 쓰려 함 예측이 거의 다 빗나간다. 실제 함정은 돌려봐야 안다 실수 발생 시 즉시 반영. 살아있는 문서.
CLAUDE.md 100줄 초과 방치 매 step 토큰이 비싸진다 주제별로 분할 → 참조 한 줄만 남겨라
created_at, started_at을 손으로 적어둠 execute.py가 덮어쓴다. 의미 없는 일. 비워둬라
docs/ 안의 placeholder를 그대로 두기 가드레일이 빈 텍스트만 주입됨 — 무방비 실제 프로젝트 내용으로 채워라
error 상태인 채로 다음 phase 추가 execute.py가 시작 시점에 차단한다 (_check_blockers) 먼저 error를 복구하라
completed 처리 후 contract 안 적기 다음 step에 긴 summary가 그대로 누적된다 1~3줄 contract를 챙겨 써라
