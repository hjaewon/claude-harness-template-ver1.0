01하네스가 뭐예요?

하네스(Harness)는 '말에 채우는 마구'라는 뜻. 큰 짐승(=AI)을 잘 다루기 위한 도구다.

AI에게 "쇼핑몰 하나 만들어줘"처럼 큰 작업을 한 번에 시키면 결과가 엉망이 된다. 한 번에 처리할 수 있는 양이 정해져 있고, 중간 실수에도 멈추지 않고, 처음 받은 규칙도 점점 잊기 때문이다.

그래서 우리 팀이 만든 게 하네스 템플릿이다. 큰일을 작은 단계로 잘게 잘라서 순서대로 시키는 자동화 도구. 각 단계마다 잘 됐는지 자동으로 검증하고, 실패하면 알아서 다시 시킨다.

02핵심 비유

이 비유 하나만 머리에 박아두면 나머지는 다 따라온다.

analogy

외주 개발자 5명에게 순서대로 일을 시키는 PM

당신은 PM이고, 외주 개발자 5명이 옆에 대기 중이다. 다만 이들은 서로 모르고, 다음 사람은 이전 사람이 무엇을 했는지 본인이 직접 코드를 봐야 안다.

5명에게 똑같이 큰 명세서를 줘봤자 다 비슷한 코드를 만들거나 충돌한다. 그래서 다음과 같이 한다:

  1. 일을 5단계로 잘게 쪼갠다 (테이블 → 함수 → 화면 → 테스트 → 통합).
  2. 단계마다 개발 지시서(step.md)를 한 장씩 써둔다.
  3. 회사 사규집(CLAUDE.md)을 매번 같이 동봉한다. 특히 "지난번에 자주 실수했던 함정"을 누적해서 적어둔다.
  4. 첫 번째 개발자에게 1단계 지시서를 주고, 작업이 끝나면 검수한다.
  5. 두 번째 개발자에게 2단계 지시서 + "1단계는 이런 결과물이 나왔다"는 한 줄짜리 인수인계서를 준다.
  6. 5단계가 끝날 때까지 반복.

하네스 템플릿이 하는 일이 정확히 이거다. 외주 개발자가 사람이 아니라 Claude(AI)고, 각 단계 사이를 자동으로 이어주는 조교 스크립트(execute.py)가 거의 모든 잡일을 처리한다.

SAP 비유 ABAP 큰 프로젝트를 발주할 때 한 장짜리 메가 명세서 대신, 개발지침서 + 단위테스트 명세서 + 트랜스포트별 작업 분할표로 단계별 검수받는 것. 그걸 AI 버전으로 자동화한 것.

03전체 그림

큰 그림부터 본다. 세부는 뒤에서 푼다.

   [PM = 당신]           [조교 = execute.py]        [외주 = Claude]

   ① 사규집 + 매뉴얼 작성
      (CLAUDE.md, docs/)
                           ──▶  매 단계 동봉           ──▶  지시서+규칙 읽고
                                                          코드 작성
   ② 5단계로 쪼개기
      (step0.md ~ step4.md)
                           ──▶  1단계 지시서 전달
                                                          ──▶  AC 검증 (자기 시험)
                                                                통과 → 다음으로
                                                                실패 → 다시 시도
                           ◀──  결과 + 인수인계서
                           ──▶  2단계로 진행
                                ...

   ③ 끝나면 PM이 검토
   ④ 실수 발견 시 → 사규집 업데이트  ← 시간이 갈수록 품질이 오르는 핵심 루프

사람이 하는 일은 ①·②·④. 나머지는 조교가 알아서 돈다.

04폴더 구조

하네스 템플릿을 받으면 다음 구조가 들어있다. 각 폴더가 어떤 방인지 보면 외울 필요가 없다.

claude-harness-template/         ← 사무실 전체
├── CLAUDE.md                    ← 사규집 (Claude가 매 작업 전 자동으로 읽음)
├── .claude/
│   ├── CLAUDE-EDITING-RULES.md  ← 사규집 편집 규칙 (Claude가 사규집 고칠 때 본다)
│   └── commands/                  · /harness, /review 같은 단축 명령
├── docs/                        ← 자료실 (매 작업에 동봉됨)
│   ├── PRD.md                         · 무엇을 만들 것인가
│   ├── ARCHITECTURE.md                · 어떻게 구조화할 것인가
│   ├── ADR.md                         · 왜 이렇게 정했는가
│   └── UI_GUIDE.md                    · 디자인 규칙
├── scripts/                     ← 조교실
│   └── execute.py                     · 조교 본체 (우리는 명령만 내림)
└── phases/                      ← 작업 지시서 보관함 (우리가 채울 곳)
    └── 0-mvp/
        ├── index.json                 · 단계별 상태
        ├── step0.md                   · 1단계 지시서
        └── step1.md                   · 2단계 지시서

핵심만 기억하면:

  • CLAUDE.md + docs/ = 매번 AI에게 같이 보내는 회사 매뉴얼
  • scripts/execute.py = 조교. 직접 손댈 일 없음.
  • phases/ = 우리가 작업할 때마다 새 폴더를 만들어 지시서를 채워넣는 곳.

05CLAUDE.md, 어떻게 채우나

CLAUDE.md는 사람용 README가 아니라 AI용 가드레일이다. AI가 매 작업 전 자동으로 읽는 행동 지침.

흔한 오해 — 사람들은 보통 이렇게 잘못 쓴다

## 기술 스택
- Next.js 15
- TypeScript strict mode
- Tailwind CSS

## 폴더 구조
- src/app/        ← 페이지
- src/components/ ← UI 컴포넌트
- src/lib/        ← 유틸리티

## 사용 라이브러리
- axios, date-fns, zustand, react-hook-form, ...

## 일반 가이드
- 테스트를 잘 작성해주세요.
- 코드를 깔끔하게 짜주세요.
- 보안에 신경 써주세요.

→ 전부 안티패턴.

  • 기술 스택 / 폴더 구조 / 라이브러리 목록 — 코드와 package.json 보면 1초에 안다. 매 작업에 동봉되니 토큰 낭비.
  • "잘 작성해주세요" — 추상적이라 AI 행동을 바꾸지 못한다.

이렇게 써야 한다

## 함정 (Claude가 자주 빠지는 곳)
- 테스트는 vitest만. jest 시도 금지.
  이유: 환경에 jest 미설치.
- DB 마이그레이션 자동 생성 금지. 무조건 수동 SQL.
  이유: 2025-11 ORM 자동 생성이 prod 인덱스를 잘못 만든 사고.
- API 호출은 src/services/ 경유. 컴포넌트에서 직접 fetch 금지.
  이유: SSR 캐시 우회로 결제 정보가 다른 유저에게 노출된 사고.

→ 형식은 "X 하지 마라. 이유: Y".

  • 코드만 봐선 절대 모를 함정·금지·도메인 트랩만 적는다.
  • "이유"가 빠진 규칙은 시간이 지나면 판단 불가능해진다. 반드시 같이 적는다.

가장 중요한 한 가지 — "실수 즉시 반영"

최고 ROI 워크플로우 CLAUDE.md를 처음부터 완벽하게 쓰려고 하지 마라. 그건 불가능하다.
대신 AI가 실수하는 바로 그 순간에 이렇게 시켜라: 
"이런 실수 다시 하지 마. 이걸 CLAUDE.md에 반영해줘."
CLAUDE.md는 사고일지처럼 키워가는 문서지, 한 번에 다 쓰는 문서가 아니다.

06일하는 순서 (A~E)

새 기능을 추가할 때마다 따르는 5단계.

A
탐색
B
논의
C
단계 설계
D
지시서 작성
E
실행

A. 탐색 — 매뉴얼 먼저 읽기

AI에게 docs/의 PRD, ARCHITECTURE, ADR을 읽으라고 시킨다. 회사가 뭘 만들려고 하고, 어떤 규칙 위에서 만드는지 AI가 파악해야 헛소리를 안 한다.

B. 논의 — 헷갈리는 건 코딩 전에 정한다

"비밀번호 어떻게 암호화할 거예요?" 같은 결정 사항을 코드 짜기 전에 합의한다.

C. 단계 설계 — 5개로 쪼개기

AI가 "이 기능은 5단계로 나누면 좋겠다"고 초안을 제시. 사람이 보고 OK/수정.

D. 지시서 작성 — 파일로 박제

사람이 승인하면 AI가 자동으로 phases/ 안에 지시서 파일들을 만든다.

E. 실행 — 조교에게 맡기기

python3 scripts/execute.py 0-mvp

이제 사람은 진행 표시기를 구경한다.

숨겨진 F단계 A~E가 끝나면 회고가 한 번 더 있다: 작업 중 AI가 같은 실수를 두 번 이상 했다면 즉시 CLAUDE.md에 반영한다. 이게 빠지면 다음 작업에서 같은 실수를 또 한다.

07왜 잘게 쪼개나

한 번에 시키면 안 되는 이유.

잘못된 방식

"쇼핑몰 만들어줘. 회원가입, 상품, 장바구니, 결제 다."

→ 큰 그림은 잡지만 디테일이 엉성하고, 잊은 게 많고, 동작 안 하는 부분이 섞인다.

하네스 방식

step0: DB → step1: 회원 → step2: 상품 → step3: 장바구니 → step4: 결제 → step5: 통합 테스트

→ 각 단계마다 검증을 통과해야 다음으로. 한 단계가 망가져도 그 단계만 다시.

단계 설계 시 꼭 지켜야 할 세 가지:

  1. 한 단계에 한 가지 일만. "DB+API+화면 다"는 안 됨.
  2. 지시서는 그 자체로 완결. "지난번에 얘기한 그거"라고 쓰면 안 됨. AI는 매번 처음 보는 사람이라 가정.
  3. 완료 기준은 실행 가능한 명령어로. "잘 동작해야 한다"는 안 됨. npm test 같은 자동 검증 명령으로.

08만들어지는 파일 3종

단계 설계가 끝나면 다음 3가지 파일이 만들어진다.

① phases/index.json — 전체 진행 현황표

{
  "phases": [
    { "dir": "0-mvp", "status": "pending" }
  ]
}

② phases/{작업명}/index.json — 이 작업의 상세표

{
  "project": "my-app",
  "phase": "0-mvp",
  "steps": [
    { "step": 0, "name": "project-setup", "status": "pending" },
    { "step": 1, "name": "core-types",    "status": "pending" },
    { "step": 2, "name": "api-layer",     "status": "pending" }
  ]
}

상태(status) 4가지

상태의미
pending아직 실행 안 함
completedAC 검증 통과
error3번 다시 해도 실패
blocked사람 도움 필요해서 멈춤

③ phases/{작업명}/stepN.md — 각 단계의 작업 지시서

# Step 1: core-types

## 읽어야 할 파일
- /docs/ARCHITECTURE.md          ← 회사 설계 매뉴얼
- /docs/ADR.md                   ← 기술 선택 이유
- src/lib/db.ts                  ← 이전 단계가 만든 파일

## 작업
src/types/user.ts에 다음 타입을 정의한다.
  · id는 자동 생성 (UUID v4)
  · email은 소문자로 정규화
  · 외부 라이브러리 추가 금지

## Acceptance Criteria      ← 자기 검증 명령어
  npx tsc --noEmit
  npm test -- types

## 금지사항
- uuid 라이브러리 설치 금지. 이유: ADR-002 외부 의존성 최소화.

09인수인계서 (contract)

한 단계가 끝나면 AI가 두 가지를 적어둔다.

이름누가 읽나길이
summary나중에 사람이 검토할 때길어도 됨
contract다음 단계 AI1~3줄

매번 긴 요약을 다음 AI에 보내면 토큰이 폭발한다. 그래서 다음 AI에겐 짧은 인수인계서만.

{
  "step": 7,
  "name": "rule-engine",
  "status": "completed",
  "summary": "승인된 YAML 규칙만 ERROR/WARN을 만드는 룰 엔진 구현.
              paths.py에 dotted-path 해석기 추가, 테스트 12개 통과. 설계 근거 ADR-003.",
  "contract": "engine.execute_rules() 함수 추가. eval/exec 사용 금지 유지."
}

10조교(execute.py)가 해주는 일

우리는 한 줄 명령만 친다. 뒤에서 조교가 자동으로 처리한다.

① 작업 공간 분리

별도 작업 공간을 자동으로 만든다. 원본 코드는 안 건드린다.

② 사규집 자동 첨부

AI를 깨울 때마다 CLAUDE.md + docs/를 통째로 같이 보낸다.

③ 이전 단계 인수인계

완료된 단계의 contract를 다음 AI에 전달.

④ 자가 교정 (재시도)

실패 시 최대 3번 재시도. "이전에 이런 에러 났다"는 메시지를 같이 보낸다.

⑤ 자동 commit

각 단계 끝나면 코드 commit과 관리 정보 commit을 따로 만든다.

⑥ 시간 기록

각 단계 시작/종료/실패 시간을 자동 기록.

사규집을 짧게 유지해야 하는 이유 ②번 때문이다. 매 단계마다 CLAUDE.md + docs가 통째로 동봉된다. 100줄짜리면 100줄이 매번, 1000줄짜리면 1000줄이 매번. 단계 5개면 5배. 그래서 100줄 이하 유지가 중요하다.

11막혔을 때

두 가지 케이스만 알면 된다.

케이스 1: error (3번 시도해도 실패)

  1. phases/{작업명}/index.json 파일을 연다.
  2. 실패한 단계의 error_message 항목을 읽는다.
  3. 지시서가 잘못됐으면 지시서를 고친다. 진짜 코드 문제면 직접 손본다.
  4. 같은 실수가 반복되는 패턴이면, CLAUDE.md에 함정 노트로 박는다.
  5. status를 "pending"으로 바꾸고 error_message 삭제.
  6. 명령어 다시 실행.

케이스 2: blocked (사람 도움 필요)

  1. blocked_reason을 읽고 필요한 것 확인 (API 키 발급, 외부 가입 등).
  2. 사람이 직접 처리.
  3. status 되돌리고 재실행.
하지 마라 에러난 단계를 그냥 completed로 바꿔서 넘어가는 것. 하네스의 가치는 단계마다 검증이다. 우회하면 의미가 없다.

매 사이클마다 이 회고가 반복되면서 사규집이 자란다. 그게 진짜 가치다.

13자주 빠지는 함정

처음 1~2주 안에 거의 모두가 한 번씩 밟는 함정.

실수왜 안 되나올바른 방향
한 단계에 너무 많은 일 (백엔드+화면+테스트 다) AI가 우선순위를 잃는다 단계를 더 잘게 쪼개라
AC를 "잘 동작해야 한다"로 적음 AI가 자기가 잘한다고 우기고 완료 처리 실행 명령어로 적기 (npm test 등)
step.md에 "지난번 우리가 얘기한 것처럼" AI는 매번 처음 보는 사람이라 그 얘기를 모른다 관련 파일 경로를 직접 적기
CLAUDE.md를 처음부터 완벽하게 쓰려 함 실제 함정은 돌려봐야 안다 실수 발생 즉시 반영. 살아있는 문서로 키워라.
CLAUDE.md에 라이브러리 전체 목록 나열 package.json 보면 안다. 토큰 낭비. 핵심 프레임워크만 5줄 이내
CLAUDE.md 100줄 넘어가도 방치 매 단계 토큰이 비례해서 비싸진다 주제별로 분할 → 참조 한 줄만 남기기
docs/ 안의 placeholder를 그대로 둠 가드레일이 빈 텍스트로 동봉됨 실제 프로젝트 내용으로 채우기
error 상태인데 그대로 두고 새 작업 시작 조교가 시작 시점에 막아버린다 먼저 error 단계 복구
기억할 두 가지 ① 좋은 step.md를 쓰는 것이 일의 절반. ② 나머지 절반은 CLAUDE.md를 사고일지처럼 키우는 것. 두 가지만 잘하면 하네스는 알아서 돈다.
